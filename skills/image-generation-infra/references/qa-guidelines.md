# QA Guidelines

QA is split into two categories with different executors, and the split is load-bearing: deterministic checks are cheap, objective, and automatable, so they run on every output and can trigger automatic retries. Visual checks require model or human judgement, so they are evaluated explicitly and their verdicts are recorded — never silently assumed.

## Deterministic checks (automatable)

Run these with scripts on every generated output:

- Output file exists at the declared `output.path`.
- Filename stem matches the asset ID.
- Format matches the declared format (magic bytes, not just extension).
- Dimensions match `generation.size` (or aspect ratio within tolerance).
- Expected number of files exists (variants, sprite-sheet outputs).
- Alpha channel exists when `background: transparent` is declared.
- Output is inside `project.output_root`; no path escaped the intended directory.
- The manifest entry has a valid design-sheet path.
- All referenced files exist.
- No duplicate asset IDs in the manifest.

A deterministic failure is a hard failure: mark `qa_failed`, record the reason in `failed-assets.json`, and retry automatically up to `max_attempts`.

`scripts/validate_manifest.py` covers the manifest-side checks before a run; `scripts/find_missing_assets.py` diffs manifest against disk after a run.

## Visual checks (model or human judgement)

Evaluate these against the asset's `qa.must_have` / `qa.must_not_have` lists, the subject's design sheet, and `style.md`:

- Subject matches the canonical reference (same character, not a cousin).
- Character proportions are consistent with the design sheet.
- Orientation is correct.
- Pose is readable at the intended display size.
- Frame sequence is coherent (sprite sheets: consistent identity, spacing, baseline across frames).
- Style matches the project (rendering, line treatment, lighting).
- Palette stays within the intended visual language.
- No important content is cropped.
- Text, when declared, is exactly the declared string and legible.
- No prohibited elements from the negative rules appear.

Record every visual verdict with a reason. An automated visual pass may retry obvious failures (extra limbs, wrong orientation), but the workflow must never claim that subjective visual consistency is guaranteed — final identity judgement belongs to a human reviewer, and `approved` status should normally reflect human sign-off.

## Retry policy

- Retries are budgeted per asset by `max_attempts` (default from `defaults.max_attempts`).
- Each retry records what failed and what changed (`retry_reason`, prompt adjustment) in `retry-assets.yaml` — retrying an identical prompt and hoping is a last resort, not a strategy.
- Assets that exhaust their budget go to `qa_failed` with `retry_decision: rejected` or wait for human triage; they are never silently dropped.
- Approved assets are never regenerated by a retry pass.

## Writing good per-asset QA rules

`qa.must_have` / `qa.must_not_have` entries should be checkable by looking at the image:

- Good: "feet aligned to the expected baseline", "no additional weapons", "text reads exactly 'PLAY'".
- Bad: "looks good", "high quality", "consistent" (consistent with what? — name the reference).

Global rules live in `qa-checklist.md` and `negative-prompt-rules.md`; per-asset rules cover only what is specific to this asset. If the same must_not_have appears on many assets, promote it to the global negative rules.
